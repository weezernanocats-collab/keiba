# 競馬予測システム v6.0 統合改修プラン

**現状（v5.3 リーケージ修正後）**: 単勝26.0%, 複勝55.2%, ROI 72.9%
**XGBoost**: NDCG@1 0.4994, Top-1 33.0%, Top-3 66.4% (44特徴量, 85,892サンプル)
**信頼度80%+**: 単勝39.9%, 複勝71.0% (1,343件)

> **原則**: モデル層（確率精度）と意思決定層（資金管理）は完全分離する。
> ROIの前にBrier Score/BSSで確率精度を定点観測し、施策の有効性を数理的に検証する。

---

## Phase 0: 評価基盤整備（最優先・全施策の前提）

> 適切なスコアリングルール（Proper Scoring Rules）なしに改善を測定することは不可能。
> この基盤がないと、以降の施策が「効いたか」「過学習か」を判断できない。

### 0-1. Brier Score & BSS 導入 ★必須
**想定効果**: 精度向上ではなく「精度測定能力」の獲得
**工数**: 4-6時間

**現状の問題**:
- 的中率（26.0%）は「当たったか外れたか」の粗い二値指標
- モデルが出力する確率の「質」を測定する手段がゼロ
- 施策を追加しても本当に改善したか判断できない

**実装内容**:
```
Brier Score = (1/N) × Σ(predicted_prob - actual_outcome)²
  - predicted_prob: 各馬のsoftmax正規化勝率（スコア→確率変換）
  - actual_outcome: 1(勝ち) or 0(負け)

BSS = 1 - (BS_model / BS_baseline)
  - BS_baseline = 1/fieldSize（均等確率）のBrier Score
  - BSS > 0 ならモデルに価値あり。BSS < 0 なら使い物にならない

市場BSS = 1 - (BS_model / BS_odds)
  - BS_odds = オッズから逆算した市場確率のBrier Score
  - 市場BSS > 0 なら市場を上回っている（＝長期利益の必要条件）
```

**対象ファイル**: `accuracy-tracker.ts`, `prediction-engine.ts`
- predictions テーブルに per-horse win probability を保存
- evaluateRacePrediction() でBrier Scoreを算出・記録
- getAccuracyStats() でBSS（対均等・対市場）を返す

### 0-2. キャリブレーションプロット & ECE ★必須
**想定効果**: 過信/過小評価の可視化
**工数**: 3-4時間

**現状の問題**:
- 信頼度4バンド（15-39, 40-59, 60-79, 80-100）は粗すぎる
- モデルが「30%で勝つ」と言った馬が実際に30%勝っているか不明
- 本命馬の過大評価・大穴の過小評価が定量化されていない

**実装内容**:
```
ECE (Expected Calibration Error):
  = Σ (|bin_size/N| × |avg_predicted_prob - avg_actual_win_rate|)
  10ビン: 0-10%, 10-20%, ..., 90-100%

キャリブレーション階層（Van Calster分類）:
  Mean: 全体平均勝率 ≈ 1/fieldSize → 基本チェック
  Weak: 傾き≈1, 切片≈0 → 系統的バイアスの検出
  Moderate: ビン内一致 → 実用レベルの確認
```

**対象ファイル**: `accuracy-tracker.ts`, admin UI
- 10ビン分割でECEを算出
- /admin にキャリブレーションプロット表示

### 0-3. 対数損失（Log Loss）追加
**想定効果**: 過信ペナルティの定量化
**工数**: 2時間

```
LogLoss = -(1/N) × Σ [y×log(p) + (1-y)×log(1-p)]
  - 「勝率99%」が外れた場合に巨大ペナルティ → 過信を検出
  - Brier Scoreと併用で多角的評価
```

### 0-4. Platt Scaling（確率較正）
**想定効果**: Brier Score 10-20%改善（先行研究で0.495→0.35の事例あり）
**工数**: 6-8時間

**現状の問題**:
- XGBoostランカーの出力は「順序スコア」であり確率ではない
- レース内softmaxは近似だが、較正されていない
- confidence（15-92）はスコア差ベースで確率的意味がない

**実装内容**:
```
Train/Calibration/Test 3分割:
  1. Training (70%): XGBoostランカーを学習
  2. Calibration (15%): Platt Scaling (ロジスティック回帰) or Isotonic Regression
     → モデルの生スコアを実確率へマッピング
  3. Test (15%): 較正済みモデルの最終評価

較正後:
  - per-horse win_probability が実際の勝率と一致
  - confidence = calibrated probability (意味のある確率値)
```

**対象ファイル**: `train_model.py`, `ml-client.ts`, `prediction-engine.ts`

## Phase 1: 特徴量改善（既存データ活用・低リスク）

> DBに既存のデータから新特徴量を抽出。スクレイパー変更不要。

### 1-1. オッズをXGBoost訓練から除外 → バリューフィルターへ
**想定効果**: 単勝+1-2%, ROI+5-8%
**工数**: 4-6時間

**現状の問題**:
- marketOdds (重み3%) + odds + oddsLogTransform が訓練特徴量に含まれる
- モデルが「市場と同じ予測」を学習してしまい、独自のエッジが生まれない
- umaro_ai事例: オッズ除外で独自予測→市場との乖離でROI 141%達成

**実装内容**:
```
訓練時: marketOdds, odds, oddsLogTransform を特徴量から除外
推論時: model_prob vs implied_prob(オッズ逆算) の乖離 = Value Score
  value = model_prob / implied_prob
  value > 1.2 → 過小評価馬（バリュー）
  value < 0.8 → 過大評価馬（見送り）
```

### 1-2. 騎手乗り替わりシグナル（ΔJockey）
**想定効果**: 単勝+0.5-1%
**工数**: 3-4時間

**現状の問題**:
- 騎手の能力は評価しているが「乗り替わり」のシグナルを無視
- 「リーディング騎手への乗り替わり」は陣営の勝負気配を示す強力な手がかり

**実装内容**:
```
past_performances から前走の騎手IDを取得
jockeySwitch = (前走騎手 ≠ 今走騎手) ? 1 : 0
jockeySwitchQuality = 今走騎手WR - 前走騎手WR
  正値 → 格上騎手への乗り替わり = ポジティブシグナル
  負値 → 格下騎手への乗り替わり = ネガティブシグナル
```

### 1-3. コーナー通過順位差（ペース加速指標）
**想定効果**: 単勝+0.5-1%
**工数**: 3-4時間

**現状の問題**:
- corner_positions (例: "2-3-3-2") が過去走にあるが、脚質分類にのみ使用
- 「3コーナー→4コーナーで何頭まくったか」= 末脚の強さの直接指標

**実装内容**:
```
cornerDelta = 第3コーナー順位 - 第4コーナー順位  # 正値=加速
avgCornerDelta = 直近5走の平均
cornerAcceleration: XGBoost特徴量として追加
```

### 1-4. 交互作用特徴量（6個追加）
**想定効果**: 単勝+1-1.5%
**工数**: 3時間

```
追加特徴量:
  weight × speedRating           → パワー指標（重い斤量でも速い馬）
  age × distance                 → 3歳×中距離の相性
  jockeyAbility × recentForm     → 騎手力×馬の調子の相乗効果
  fieldSize × postPosition       → 多頭数での外枠不利を定量化
  rotation × recentForm          → 休み明け×調子の相互作用
  trackCondition × sireAptitude  → 重馬場×血統の適性
```

### 1-5. 着差の定量化強化
**想定効果**: 単勝+0.3-0.5%
**工数**: 2-3時間

**現状の問題**:
- marginCompetitiveness（重み1%）は着差頻度のみ
- 着差の「量」（クビ差 vs 大差）を区別していない

```
追加: avgMarginWhenWinning (勝ち時の平均着差 → 圧勝力)
追加: avgMarginWhenLosing  (負け時の平均着差 → 接戦力)
追加: marginTrend          (直近5走の着差推移 → 上昇/下降)
```

### 1-6. 休養日数の連続値化 + 最適ローテーション
**想定効果**: 単勝+0.3-0.5%
**工数**: 2時間

**現状の問題**:
- rotation はカテゴリカルスコア（0-100）に変換されている
- 「中2週」と「中3週」の微妙な差がスコアに反映されない

```
追加: daysSinceLastRace (連続値、XGBoostが自然に分岐)
追加: optimalRotationDiff = |実ローテ - 馬の過去最高着順時のローテ|
```

### 1-7. postPositionBias / historicalPostBias 統合
**想定効果**: ノイズ低減（精度+0.1-0.2%）
**工数**: 1時間

**現状の問題**:
- 2つの枠順ファクター（各3%）が同じデータソースから計算 → 冗重
- 合計6%の重みが実質同一シグナルに費やされている

```
統合: postPositionBias に一本化（重み4%）
削除: historicalPostBias
空いた2%を他の有力ファクターに再配分
```

## Phase 2: MLモデル改善（高インパクト）

### 2-1. CatBoost Ranker 導入（XGBoost代替/併用）
**想定効果**: NDCG@1 +2-5%（0.50→0.52-0.55）, 単勝+2-3%
**工数**: 8-12時間

**根拠**: 2024年韓国競馬研究でCatBoostがXGBoostを上回る結果

```
CatBoostの優位性:
  - カテゴリ変数（騎手名、競馬場、血統等）のネイティブ処理
  - Ordered Target Encoding（リーケージリスク低減）
  - 対称木構造 → 推論速度が速い

比較実験:
  XGBRanker (現行) vs CatBoostRanker vs LightGBMRanker
  全て同一データ・同一特徴量で NDCG@1, Brier Score, BSS を比較
```

### 2-2. スタッキングアンサンブル
**想定効果**: 単勝+1.5-2.5%, Brier Score -5-10%
**工数**: 10-15時間

**現状の問題**:
- ルールスコア(60%) + XGBoost(40%) の固定ブレンド
- 最適な混合比率はレース条件によって変わるはず

```
Level-0 (ベースモデル):
  1. XGBRanker (現行)
  2. CatBoostRanker (新規)
  3. ルールベース19因子スコア
  4. LightGBM binary:logistic (勝率直接予測)

Level-1 (メタ学習器):
  - Logistic Regression / Ridge Regression
  - 入力: 4モデルの出力確率 + レースコンテキスト(距離,馬場,頭数)
  - 出力: 較正済み勝率

重要: Level-1の学習にはLevel-0の学習に使っていないデータを使用（リーケージ防止）
```

### 2-3. カテゴリ別専門モデル
**想定効果**: 単勝+1-2%
**工数**: 8-10時間

```
5カテゴリ（既存のcategorizeRace()を活用）:
  turf-sprint  (~1400m芝)
  turf-mile    (1500-1800m芝)
  turf-long    (1900m+芝)
  dirt-short   (~1600mダ)
  dirt-long    (1700m+ダ)

各カテゴリ専用XGBoost/CatBoostを学習
カテゴリ毎にサンプル数が足りない場合 → グローバルモデルにフォールバック
```

### 2-4. エクスポート時のデータリーケージ修正
**想定効果**: モデル品質の基盤改善（間接的に+0.5-1%）
**工数**: 3-4時間

**現状の問題**:
- `export-training-data.ts` でtrainerWinRate等が全期間から算出
- XGBoost学習データにも時間方向のリーケージが残存

```
修正: 各レースのdateを基準に、それ以前のデータのみからtrainer/jockeyの統計量を算出
方法: エクスポート時にSQLクエリに WHERE r.date < ? を追加
```

### 2-5. NDCG gain関数のカスタマイズ
**想定効果**: ROI+3-5%
**工数**: 4-6時間

**現状の問題**:
- 現在のgain: {1着: 3, 2着: 2, 3着: 1, 他: 0}
- 「1着のオッズ10倍の馬を当てる」方が「1着のオッズ1.5倍」より遥かに価値が高い

```
オッズ加重gain:
  gain = relevance × log(1 + odds)
  1着×10倍 = 3 × log(11) = 7.2  →  高配当的中を重視
  1着×1.5倍 = 3 × log(2.5) = 2.7 → 本命は相対的に低く

効果: モデルが「穴馬の中から来る馬を見抜く」方向に学習
```

## Phase 4: 意思決定層（モデル層と完全分離）

> ここからはモデルの「予測精度」ではなく「賭け方の最適化」。
> Phase 0-3でBrier Score / BSSが市場を上回ることが確認された後に実装する。

### 4-1. Kelly Criterion 資金管理
**想定効果**: ROI +10-20%（精度は変わらない、資金効率が変わる）
**工数**: 6-8時間

```
f* = (b×p - q) / b
  b = オッズ - 1
  p = モデル算出の勝率（較正済み）
  q = 1 - p

実装:
  fractional Kelly (f*/4) で保守的に運用
  f* < 0 → ベット見送り（期待値マイナス）
  f* > 0.25 → 上限キャップ（過集中防止）
```

### 4-2. バリューベッティング閾値
**想定効果**: ROI +5-15%
**工数**: 4-6時間

```
valueEdge = (model_prob × odds) - 1
  > 0: 正の期待値（ベット対象）
  > 0.15: 強いバリュー（推奨）
  < 0: 負の期待値（見送り）

拡張: 馬連・ワイド・三連複への期待値計算
  馬連EV = Σ(model_prob_combo × odds_combo) - 1
```

### 4-3. モンテカルロ・シミュレーション
**想定効果**: 戦略の統計的信頼性の担保
**工数**: 8-12時間

**重要性**: ROI 72.9%が「スキル」か「分散」かを検証

```
手順:
  1. 全予測のmodel_probとoddsを取得（N=6,280レース）
  2. 10,000回のブートストラップ:
     - 各回でN件をランダムサンプリング（復元）
     - Kelly/Flat/Value各戦略でROIを算出
  3. 出力:
     - ROIの95%信頼区間
     - 破産確率（Kelly戦略で残高0になる確率）
     - 必要サンプル数（ROI > 100%を95%信頼度で示すには何レース必要か）

P値 < 0.05 なら「スキルによる利益」
P値 > 0.05 なら「運の可能性が排除できない」
```

### 4-4. アンチパターン学習（過大評価本命の回避）
**想定効果**: ROI +3-5%
**工数**: 6-8時間

```
過去データから「model_prob高い & odds低い & 負けた」パターンを分析
  → 過大評価の共通特徴（例: 休み明け本命、初距離本命）を抽出
  → 該当パターンのベット倍率を自動減額
```

**想定効果**: 精度劣化の早期検出
**工数**: 4-6時間

```
月次で XGBoost feature_importances_ を記録
  → 重要度の急変 = データ分布の変化を検知
  → 自動アラート + 再学習トリガー
```

### 5-2. 近走重みの指数減衰化
**想定効果**: 単勝+0.3-0.5%
**工数**: 3-4時間

```
現在: [0.35, 0.25, 0.20, 0.12, 0.08] (ハードコード)
改善: weight = exp(-days_since_race / halflife)
  halflife = 60日（パラメータはBrier Scoreで最適化）

  bandwidth = 300m（自然な減衰）

### 5-4. キャリブレーション重みの時間減衰
**想定効果**: 最新レース傾向への追従性向上
**工数**: 2-3時間

```
現在: calibrateWeights() は全期間均等

## 推定改善幅（保守的見積もり）

| Phase | 期間 | 単勝的中率 | 複勝的中率 | Brier Score | BSS (対市場) | ROI |
|-------|------|----------|----------|-------------|-------------|-----|
| **現状 v5.3** | - | 26.0% | 55.2% | 未計測 | 未計測 | 72.9% |
| **Phase 0** (評価基盤) | 1-2週 | 26.0% | 55.2% | 計測開始 | 計測開始 | 72.9% |
| **Phase 1** (特徴量) | 2-3週 | 28-30% | 57-60% | -5~10%改善 | 正値目標 | 75-80% |
| **Phase 2** (ML改善) | 3-5週 | 30-33% | 60-63% | -10~20%改善 | +0.05~0.10 | 80-88% |
| **Phase 3** (データ拡張) | 5-8週 | 32-35% | 62-65% | -15~25%改善 | +0.08~0.15 | 85-92% |

> **注**: 精度向上は加算ではなく漸減する（diminishing returns）。
> Phase 4のROI改善は精度向上ではなく資金管理最適化による。
> ROI > 100% の達成には BSS(対市場) > 0 が必須条件。
> 各施策は独立ではなく相互に影響するため、Phase 0の計測基盤で実効性を逐次検証する。

## 検証プロトコル

各施策の実装後、以下の手順で効果を検証:

```
1. 計測（Phase 0基盤を使用）:
   - Brier Score (全体 + 信頼度帯別)
   - BSS (対均等確率 + 対市場オッズ)
   - ECE (10ビン)
   - Log Loss
   - 単勝的中率, 複勝的中率, ROI

2. 統計的有意性:
   - 前後比較: paired t-test or bootstrap confidence interval
   - BSS改善が95%信頼区間で正値 → 採用
   - BSS改善が有意でない → 却下 or 再検討

3. 過学習チェック:
   - Train/Val/Test のBrier Scoreが乖離していないか
   - 特定月に偏った改善でないか（月別分析）

4. 回帰テスト:
   - 既存の正しいケース（信頼度80%+帯の39.9%的中）が劣化していないか
```

## 付録: 現在のシステムで発見された技術的問題

| 問題 | 影響 | 対応Phase |
|------|------|----------|
| XGBoostランカー出力が確率ではない | 確率較正不可 | 0-4 |

| confidenceスコアがスコア差ベース（確率的意味なし） | 信頼度の信頼性なし | 0-4 |
| calibrateWeights()が全期間均等 | 季節変化に追従できない | 5-4 |
| avgRoi（算術平均）が誤解を招く | ROI過大表示 | 0-1 |
