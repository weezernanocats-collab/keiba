# 的中率改善プラン

**現状（2026-03-08 リーケージ修正後）**: 単勝26.0%, 複勝55.2%, ROI 72.9%

> **注記**: 旧数値（単勝47.5%等）はデータリーケージによる誤値。修正後の正しい精度を基準とする。
> Tier 1の項目1-4は既にv5.0-v5.2で実装済み。

---

## Tier 1: すぐできる改善（各2-6時間、合計+3-5%）

### 1. スピード指数の動的基準化 (+2-3%)
**問題**: 基準タイムがハードコード。東京芝1600と京都芝1600を同じ基準で比較している。
```
現在: standardTimes = { '芝': { 1600: 95.0 } }  // 全コース共通
改善: 過去100走の平均タイムをコース×距離×馬場別に動的算出
```
**効果**: 最も使われるスピード指数の精度が根本的に上がる。

### 2. 枠順バイアスをリアルタイム化 (+1.5%)
**問題**: 枠の有利不利がハードコード。`calculateTodayTrackBias()`は既に実装済みだが予測に未使用。
```
現在: biasMap = { '東京': { '芝_inner': 60 } }  // 固定値
改善: 直近30日の同コース結果から内外バイアスを自動計算
```

### 3. 騎手の直近フォーム (+1-2%)
**問題**: 騎手の通算成績は使用中だが、直近30日/年間の調子を無視。
```
追加特徴量: jockeyRecent30DayWinRate, jockeyYTDWinRate, jockeyTrend
```

### 4. ベイジアン平滑化 (+0.5-1%)
**問題**: データ不足時にスコア50（中立）を返すが、これは不正確。
```
現在: if (races < 10) return 50;
改善: adjusted = (wins + alpha) / (races + alpha + beta)  // ベータ分布で推定
```

### 5. 着差データの活用 (+1%)
**問題**: past_performances.marginに着差があるが未使用。大差勝ちの馬と接戦勝ちの馬を区別できない。
```
追加: competitiveness = 着差の大きさから「力差」を数値化
```

## Tier 2: 中規模改善（各8-30時間、合計+5-8%）

### 6. LambdaMART ランキングモデル (+3-5%) **最大インパクト**
**問題**: 現在は「勝つか負けるか」の二値分類。2着と10着を同じ「負け」として扱っている。
```
現在: XGBoost binary:logistic → 勝つ確率
改善: LightGBM rank:ndcg → 着順予測（NDCG@3最適化）
```
ランキングモデルは「どの馬がどの馬より前に来るか」を直接最適化する。

### 7. 距離×馬場別の専門モデル (+2-3%)
**問題**: 1モデルで短距離芝もダート長距離もカバー。特徴量の重要度が混ざる。
```
改善: Sprint(~1400m)/Mile(1500-1800m)/Classic(1900-2400m)/Stayer(2500m+)
      × 芝/ダート = 最大8モデル
```

### 8. 交互作用特徴量 (+1.5-2%)
```
追加:
- handicapWeight × speedRating  → パワー指標
- age × distance               → 3歳は中距離に強い傾向
- jockeyRecentForm × horseRecentForm → 勢いの相乗効果
```

### 9. コース別基準タイム (+2%)
```
追加: 10場×距離別の基準タイムテーブル
効果: 「東京は速い、小倉は時計がかかる」を定量化
```

### 10. 調教師の得意距離/グレード (+0.8%)
```
追加: trainerSprintWR, trainerClassicWR, trainerGradeWR
効果: 「短距離得意な厩舎」「重賞に強い厩舎」を定量化
```

## Tier 3: 高度な改善（各10-20時間）

### 11. 近走の重み付けを指数減衰に
```
現在: [0.35, 0.25, 0.20, 0.12, 0.08]  // ハードコード
改善: weight = exp(-days_ago / halflife)  // 日数ベースの自然な減衰
```

### 12. 距離適性をRBFカーネルで連続化
```
現在: 「±100m以内」「±200m以内」のバイナリ
改善: similarity = exp(-(distance_diff)^2 / bandwidth)  // 連続的な類似度
```

### 13. 厩舎コメントのNLP分析
```
horses.training_commentにデータあり（未使用）
キーワード: 「順調」「キレ」→プラス / 「疲労」「故障」→マイナス
```

### 14. 母父の血統分析
```
現在: father_nameのみ使用
追加: mother_name → 母父の適性（スタミナ、スピード）
```

## おすすめ実装順

1. **スピード指数の動的化** ← 最も費用対効果が高い
2. **枠順バイアスのリアルタイム化** ← 既存関数の活用で簡単
3. **LambdaMART導入** ← 最大インパクト（ただし工数大）
4. **騎手直近フォーム** ← データ既存、追加クエリのみ
5. **交互作用特徴量** ← XGBoost再学習だけで効果
