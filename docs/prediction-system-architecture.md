## 1. データ収集 (Data Collection)

### 1.1 データソース

本システムのデータは全て **netkeiba.com** からスクレイピングで取得する。

| データ種別 | 取得元 | 取得タイミング | 格納先テーブル |
|---|---|---|---|
| レース一覧 | `race.netkeiba.com/top/race_list_sub.html` | 朝 09:00 JST | `races` |
| 出馬表（馬・騎手・枠順） | `race.netkeiba.com/race/shutuba.html` | レース一覧取得後 | `race_entries` |
| オッズ（単勝・複勝） | `race.netkeiba.com/api/api_get_jra_odds.html` (JSON) | 出馬表取得後 / 手動 | `odds` |
| 馬の詳細（血統・戦績） | `db.netkeiba.com/horse/{horseId}` | 出馬表取得後 | `horses`, `past_performances` |
| レース結果（着順・タイム） | `race.netkeiba.com/race/result.html` | 17:00 全レース終了後 / 再生成時 | `race_entries` (result_* 列) |

### 1.2 現在のデータ蓄積量

| テーブル | 件数 | 期間 |
|---|---|---|
| races (レース) | ~6,310 | 2024-05-04 〜 現在 (約1年10ヶ月) |
| race_entries (出走馬) | ~86,644 | 同上 |
| horses (馬マスター) | ~16,312 | - |
| past_performances (過去成績) | ~69,504 | 2017-12-09 〜 現在 (約8年分) |
| odds (オッズ) | ~86,000 | カバー率 98.7% |
| jockeys (騎手マスター) | ~197 | - |
| predictions (予測) | ~6,300 | - |
| prediction_results (評価結果) | ~6,280 | - |

レースデータは約1年10ヶ月分蓄積。各馬の過去成績（past_performances）は馬詳細ページから取得するため、最大約8年前まで遡った走歴データを保持している。

### 1.3 取得フロー

```
Vercel Cron (vercel.json: 2ジョブ)
│
├─ 09:00 JST (0 0 * * *) ── bulk_chunked トリガー
│    └─ /api/sync にPOST → チェーン実行:
│       dates → race_details → horses → results → odds → predictions → evaluate
│
└─ 17:00 JST (0 8 * * *) ── 結果取得 + 予想照合
     └─ scheduler.ts の runCronJob() を実行
```

各リクエストには **1.2秒間隔のレートリミット**、**10秒タイムアウト**、**最大2回リトライ**（指数バックオフ）を適用し、取得元サーバーに過度な負荷をかけない設計としている。

### 1.4 バルクインポート

過去データを一括取り込みする機能。日付範囲を指定し、レース一覧→出馬表→馬詳細→オッズ→結果→予想生成までを自動実行する。Vercelの60秒タイムアウト制約に対応するため、チャンク単位で分割処理し、APIを繰り返し呼び出して段階的に完了させる。

### 1.5 手動オッズ取得

レース詳細ページに「オッズ取得/更新」ボタンを配置。`POST /api/odds` でnetkeiba APIから単勝・複勝オッズを取得する。レース前の時間帯（netkeiba APIが `status: "middle"` を返す期間）は取得不可。

### 1.6 netkeiba スクレイピング注意点

| 項目 | 注意点 |
|---|---|
| エンコーディング | `race_list_sub.html` = UTF-8、その他 = EUC-JP |
| トラックタイプ | 出馬表では省略形 `ダ`（`ダート`ではない） |
| 馬場状態 | `稍`（`稍重`の省略形）の場合あり |
| グレード | CSSクラスベース（`Icon_GradeType1/2/3`）、テキストではない |
| 騎手・調教師名 | 出馬表では截断される（netkeiba仕様） |
| オッズ | `result.html` に確定オッズ（col[10]）と人気（col[9]） |
| レートリミット | 約2,600リクエストで400エラー → 12時間以上待つ必要あり |

### 2.2 カテゴリ別ウェイトプロファイル (v5.0新規)

レース条件に応じて5カテゴリに分類し、ファクター重みに乗数を適用する。

| カテゴリ | 条件 | 主な重み調整 |
|---|---|---|
| **turf_sprint** | 芝 ≤1400m | スピード指数×1.3、枠順×1.5、上がり3F×0.7 |
| **turf_mile** | 芝 1500-1800m | ほぼ基本重みのまま（マイルが基準） |
| **turf_long** | 芝 1900m+ | 血統×1.3、脚質×1.3、安定性×1.2、枠順×0.6 |
| **dirt_sprint** | ダート ≤1400m | スピード×1.4、クラス×1.2、枠順×1.3、上がり3F×0.6 |
| **dirt_long** | ダート 1500m+ | 馬場適性×1.3、スピード×1.2、直近成績×1.1 |

乗数適用後、合計が1.0になるよう正規化して使用する。

### 2.3 ベイズ推定と動的重み調整

各ファクターのスコアは、**データの信頼度**に応じて補正される。

```
信頼度 = min(実際のデータ数 / 統計的に必要なデータ数, 1.0)

最終スコア = 事前分布（デフォルト値） × (1 - 信頼度) + 観測スコア × 信頼度
```

例：コース適性で1走しかデータがなく、必要数が3走の場合、信頼度=0.33。スコアの67%はデフォルト値（45点）、33%が実際の計算値となる。これにより、データが少ない馬の極端なスコアを防ぐ。

さらに、各ファクターの重みに信頼度を乗じる（最低20%を保証: `重み × max(0.2, 信頼度)`）。削減された分の重みは、信頼度の高いファクターに比例再配分する（**動的重み調整**）。データの多い馬では多面的に、データの少ない馬では信頼できる要素に集中して評価する。

### 2.4 ペースボーナスと加重合計

```
totalScore = Σ (各ファクタースコア × 動的調整後の重み) + ペースボーナス
```

ペースボーナスは脚質と距離の相性に応じて加減算される（短距離=逃げ先行有利、長距離=差し追込有利）。

### 2.5 当日馬場バイアス補正

`track-bias.ts` で同日・同競馬場の結果確定済みレース（3R以上）を分析し、馬場バイアスを算出する。

| バイアス | 算出方法 | 調整幅 |
|---|---|---|
| 枠順バイアス | 上位3着の内枠/外枠比率 → 期待値(0.5)からの乖離 | ±2点 |
| 脚質バイアス | 勝ち馬の第1コーナー通過順位 → 前残り/差し有利判定 | ±2点 |

信頼度（3R=0.375 〜 8R+=1.0）でスケールし、最大 ±4点の調整を適用。

**午後の予想再生成**で効果を発揮する。朝の初回予想生成時は結果確定レースがないためスキップされる。

### 2.6 期待値(EV)に基づくバリューベット検出

Softmax確率推定で各馬の推定勝率を算出し、実際のオッズと掛け合わせて期待値を計算する。

```
P(win_i) = e^(score_i / T) / Σ e^(score_j / T)    (T=8)
EV_i = P(win_i) × odds_i
```

EV > 1.15 の馬を「バリューベット」として推奨。推奨馬券はEV降順でソートされる。

### 2.7 信頼度 (Confidence) の算出

各予想には0〜100%の信頼度が付与される。以下の3要素の合計で算出する。

| 要素 | 最大配点 | 内容 |
|---|---|---|
| スコア差 | 40点 | 上位馬間のスコア差が大きいほど本命が明確→高信頼度 |
| データ充実度 | 35点 | 全馬の平均データ量（過去走数等）が多いほど高信頼度 |
| 統計コンテキスト | 15点 | コース統計・血統統計の充実度 |

## 3. XGBoost MLブレンド

### 3.1 目的

加重合計は線形結合であるため、「コース適性が高い＋距離適性も高い」場合の相乗効果のような非線形パターンを捉えられない。XGBoostの勾配ブースティング木で勝率・複勝率を学習し、加重合計とブレンドすることで精度向上を狙う。

### 3.2 特徴量 (45次元)

| グループ | 特徴量 | 次元数 |
|---|---|---|
| 19ファクタースコア | recentForm, courseAptitude, ... marketOdds | 19 |
| コンテキスト特徴量 | 頭数, オッズ, 人気, 年齢, 性別, 斤量, 枠番 | 7 |
| カテゴリエンコード | グレード, 芝/ダート, 距離, 馬場状態 | 4 |
| 派生特徴量 | log(1+オッズ), 人気/頭数 | 2 |
| v4.2追加特徴量 | 天候, 馬体重変化, 調教師勝率/複勝率 | 4 |
| 交互作用特徴量 | 種牡馬×トラック勝率, 騎手×距離勝率, 騎手×コース勝率 | 3 |
| v5.1追加特徴量 | weightStability, weightTrendSlope, weightOptimalDelta | 3 |
| v5.2追加特徴量 | marginCompetitiveness, weatherAptitude, seasonalPattern | 3 |
| **合計** | | **45** |

### 3.3 推論フロー（TypeScriptネイティブ実装）

```
prediction-engine.ts
  │
  ├─ buildMLFeatures() で45次元ベクトル構築
  │
  ├─ callMLPredict() → TypeScriptでXGBoost JSONモデルを直接走査
  │     ├─ [優先] model/xgb_ranker.json → LambdaMART/NDCG ランキングモデル
  │     │    → softmaxで確率変換 → winProb, placeProb
  │     └─ [従来] model/xgb_win.json + xgb_place.json → Classifierモデル
  │          → sigmoid → 勝率・複勝率
  │
  └─ ブレンド: 最終スコア = 加重合計 × 60% + ML勝率 × 100 × 40%
     (比率は ML_BLEND_WEIGHT 環境変数で調整可能、デフォルト0.40)
```

Python不要 — XGBoostのJSON決定木形式をTypeScriptで直接パースし、ツリー走査で推論する。これによりVercel Hobbyプランの500MB制限を回避。

**フォールバック**: Rankerモデル未配置 → Classifierモデル → どちらもなければMLスキップ、加重合計のみ使用。

## 4. Gemini テキスト強化

### 4.1 目的

### 4.2 モデル選択とフォールバック

```
GEMINI_MODELS = ['gemini-flash-latest', 'gemini-2.5-flash', 'gemini-2.5-flash-lite']
```

1. `gemini-flash-latest`（常に最新のFlashモデルを指すエイリアス）を最優先で使用
2. quota超過（429/RESOURCE_EXHAUSTED）時は次のモデルへフォールバック
3. 全モデル失敗 → テンプレートテキストで代替
4. タイムアウト: 25秒（Vercel 60秒制限対応）

### 4.3 処理フロー

```
予想エンジンの出力（数値スコア + テンプレートテキスト）
  │
  ▼
Gemini Flash API (JSON mode)
  ├─ 入力: レース条件 + 上位6頭のスコア・脚質・血統情報
  ├─ 指示: 競馬プロ解説者として分析テキストを生成
  └─ 出力 (JSON):
       ├─ summary: 総合分析 (2〜4段落)
       ├─ picks[].reasons: 馬ごとの推奨理由 (2〜3個)
       ├─ trackBias: 馬場バイアス分析
       ├─ paceAnalysis: ペース展開予想
       ├─ keyFactors: 重要ファクター (3個)
       ├─ riskFactors: 波乱要素 (2個)
       └─ betReasonings: 馬券種別ごとの根拠
```

**フォールバック**: GEMINI_API_KEY未設定・APIエラー・タイムアウト・全モデルquota超過 → テンプレートテキストがそのまま使用される。ランキングや数値は一切変更されない。

## 5. 的中率評価と自動キャリブレーション

### 5.1 評価項目

レース確定後、予想と実結果を自動照合して以下を計測する。

| 指標 | 内容 |
|---|---|
| 単勝的中率 | 本命馬が1着になった割合 |
| 複勝的中率 | 本命馬が3着以内に入った割合 |
| Top3カバー率 | 推奨上位3頭のうち、実際に3着以内に入った頭数 |
| 馬券ROI | 推奨馬券の回収率（本命馬に単勝100円想定 × 実オッズ） |
| 信頼度キャリブレーション | 「信頼度70%の予想が実際に70%当たるか」の検証 |

### 5.2 現在の精度実績（2026-03時点）

| 指標 | 全期間 (6,280件) | 信頼度帯別（80-100%帯） |
|---|---|---|
| 単勝的中率 | 47.5% | 64.3% |
| 複勝的中率 | 69.5% | 81.3% |
| ROI | 228.7% | 269.5% |

#### 信頼度帯別パフォーマンス

| 信頼度帯 | 件数 | 単勝的中率 | 複勝的中率 | ROI |
|---|---|---|---|---|
| 80-100% | 2,588 | 64.3% | 81.3% | 269.5% |
| 60-79% | 1,958 | 39.8% | 65.7% | 201.9% |
| 40-59% | 1,724 | 31.2% | 56.1% | 198.6% |
| 15-39% | 10 | 20.0% | 60.0% | 86.0% |

#### トラック別パフォーマンス

| トラック | 件数 | 単勝的中率 | 複勝的中率 |
|---|---|---|---|
| ダート | 3,258 | 47.8% | 70.2% |
| 芝 | 2,818 | 47.3% | 68.7% |
| 障害 | 204 | 45.1% | 69.6% |

### 5.3 自動キャリブレーション

評価データが **20レース以上**蓄積されると、各ファクターの重みを自動調整する。

1. 的中レースで高スコアだったファクターの重みを上げる
2. 外れレースで誤誘導したファクターの重みを下げる
3. 算出された新しい重みを自動適用
4. `calibration_weights`テーブルに履歴を保存

### 5.4 バックテストの制約

過去レースの予測を再生成する際、以下のファクターは**レース日時点のデータ**を前提に設計されているため、古いレースの再生成では時間的バイアス（look-ahead bias）が生じる。

| ファクター | 問題 |
|---|---|
| 騎手直近フォーム | 「直近30日」を現在日基準で計算 |
| 動的基準タイム | past_performancesの全データ（未来含む）から算出 |
| 枠順バイアス（実データ） | courseDistStatsに未来データが混入 |

このため、全期間での予測再生成は意味がなく、v5.0の真の精度は今後蓄積される実戦データで評価する。

`gen-predictions-optimized.ts` と `regen-v5.ts` で採用している最適化方式。

```
通常の予測生成: レースあたり ~50クエリ × 594レース = ~30,000クエリ
最適化版:       7プリロードクエリ + レースあたり ~2書き込みのみ

プリロード (7クエリ):
  1. SELECT * FROM races               → racesById (Map)
  2. SELECT * FROM race_entries         → entriesByRace (Map)
  3. SELECT ... FROM horses             → horsesById (Map)
  4. SELECT * FROM past_performances    → perfsByHorse (Map)
  5. SELECT ... FROM jockeys            → jockeysById (Map)
  6. SELECT ... FROM odds (単勝)        → oddsByRace (Map)
  7. 派生インデックス構築               → jockeyRaceHistory, topFinisherPerfs 等

SQLインターセプター:
  client.execute() をパッチし、SQL文をパターンマッチ
  → 該当するメモリキャッシュから結果を返す（Turso読み取りゼロ）
  → INSERT/DELETE/UPDATE のみ実DBにパススルー
```

実測: 594レース / 3.4分 / キャッシュヒット率 97.5%（45,896ヒット / 1,194ミス）

### 8.3 エラーハンドリング方針

本システムは**段階的フォールバック**を採用している。各レイヤーが独立して失敗してもシステム全体は動作し続ける。

| 障害 | 影響範囲 | フォールバック動作 |
|---|---|---|
| Rankerモデル未配置 | ML推論 | Classifierモデルにフォールバック |
| 全MLモデル未配置/パースエラー | ML推論 | 加重合計スコアのみで予想（ML導入前と同等） |
| 当日バイアスデータ不足 | バイアス補正 | 補正なし（加重合計のまま） |
| 動的基準タイムデータ不足 | スピード指数 | ハードコード基準タイムにフォールバック |
| コース別枠順データ不足 | 枠順バイアス | 固定biasMapにフォールバック |
| 騎手直近フォームデータ不足 | 騎手能力 | 通算成績のみで評価 |
| オッズ未取得 | 市場オッズファクター | 事前分布（50点）にフォールバック |
| Gemini APIキー未設定 | テキスト生成 | テンプレートベースの分析テキスト |
| Gemini APIエラー/quota超過 | テキスト生成 | 次のモデルへフォールバック → 全モデル失敗時はテンプレート |
| 馬の過去データ不足 | スコア精度 | ベイズ推定で事前分布にフォールバック + 動的重み再配分 |
| スクレイピング失敗 | データ取得 | リトライ (最大2回)、失敗時はスキップして次の処理へ |
| 同期処理タイムアウト | バックグラウンド処理 | 2分後に自動リセット、再実行可能 |
| Turso読み取り制限 | 全クエリ | ローカルスクリプト（プリロード方式）で回避可能 |
