## 1. データ収集 (Data Collection)

### 1.1 データソース

本システムのデータは全て **netkeiba.com** からスクレイピングで取得する。

| データ種別 | 取得元 | 取得タイミング | 格納先テーブル |
|---|---|---|---|
| レース一覧 | `race.netkeiba.com/top/race_list_sub.html` | 朝 09:00 JST | `races` |
| 出馬表（馬・騎手・枠順） | `race.netkeiba.com/race/shutuba.html` | レース一覧取得後 | `race_entries` |
| オッズ（単勝・複勝） | `race.netkeiba.com/api/api_get_jra_odds.html` (JSON) | 出馬表取得後 / 手動 | `odds` |
| 馬の詳細（血統・戦績） | `db.netkeiba.com/horse/{horseId}` | 出馬表取得後 | `horses`, `past_performances` |
| レース結果（着順・タイム） | `race.netkeiba.com/race/result.html` | 17:00 全レース終了後 / 再生成時 | `race_entries` (result_* 列) |

### 1.2 取得フロー

```
Vercel Cron (vercel.json: 2ジョブ)
│
├─ 09:00 JST (0 0 * * *) ── bulk_chunked トリガー
│    └─ /api/sync にPOST → チェーン実行:
│       dates → race_details → horses → results → odds → predictions → evaluate
│
└─ 17:00 JST (0 8 * * *) ── 結果取得 + 予想照合
     └─ scheduler.ts の runCronJob() を実行
```

各リクエストには **1.2秒間隔のレートリミット**、**10秒タイムアウト**、**最大2回リトライ**（指数バックオフ）を適用し、取得元サーバーに過度な負荷をかけない設計としている。

### 1.3 バルクインポート

過去データを一括取り込みする機能。日付範囲を指定し、レース一覧→出馬表→馬詳細→オッズ→結果→予想生成までを自動実行する。Vercelの60秒タイムアウト制約に対応するため、チャンク単位で分割処理し、APIを繰り返し呼び出して段階的に完了させる。

### 1.4 手動オッズ取得

レース詳細ページに「オッズ取得/更新」ボタンを配置。`POST /api/odds` でnetkeiba APIから単勝・複勝オッズを取得する。レース前の時間帯（netkeiba APIが `status: "middle"` を返す期間）は取得不可。

## 2. 予想エンジン (Prediction Engine)

### 2.1 16ファクタースコアリング

各出走馬に対して、以下の16要素を0〜100点で独立にスコアリングする。

#### 個別能力ファクター (7要素、合計60%)

| ファクター | 重み | 算出ロジック | 必要データ量 |
|---|---|---|---|
| **直近成績** | 18% | 直近5走の着順を新しいほど高い重みで加重平均。1着=100点、着差に応じて減衰 | 5走 |
| **距離適性** | 12% | レース距離 ±400m 以内の過去成績。距離が近いほど高評価 | 3走 |
| **スピード指数** | 10% | 上がり3Fタイム＋走破タイムから算出。馬場・距離で正規化 | 3走 |
| **コース適性** | 6% | 同一競馬場での過去成績（連対率・複勝率） | 3走 |
| **騎手能力** | 6% | 騎手の勝率・複勝率。データ不足時はデフォルト8%で代替 | - |
| **クラス実績** | 4% | 重賞・オープンなど上位クラスでの実績（勝率） | 3走 |
| **馬場適性** | 4% | 芝/ダート、良/稍重/重/不良での過去成績。道悪巧者ボーナス付与 | 2走 |

#### 走法・位置ファクター (5要素、合計25%)

| ファクター | 重み | 算出ロジック | 必要データ量 |
|---|---|---|---|
| **上がり3F** | 8% | 直近の上がり3Fタイムの平均。35秒以下で高評価 | 3走 |
| **脚質適性** | 5% | 通過順位から判定（逃げ/先行/差し/追込）。距離によるボーナスあり | 3走 |
| **安定性** | 5% | 着順の標準偏差が小さいほど高評価（安定して好走する馬を評価） | 5走 |
| **ローテーション** | 4% | 前走からの間隔を評価。最適間隔（3〜8週）で高得点。叩き良化パターン検出 | 2走 |
| **枠順バイアス** | 3% | 枠番による有利不利。コース・距離特性に応じた統計データ | 20走分の統計 |

#### 統計・血統ファクター (4要素、合計15%)

| ファクター | 重み | 算出ロジック | 必要データ量 |
|---|---|---|---|
| **父系適性** | 5% | 父馬の産駒成績を芝/ダート・距離帯・馬場状態で集計 | 産駒10走 |
| **季節パターン** | 4% | 馬ごとの月別成績パターン（「秋に強い馬」等） | 月3走 |
| **騎手×調教師** | 3% | 騎手と調教師のコンビ実績（勝率） | 5走 |
| **枠順統計** | 3% | コース×距離での枠番別勝率（統計的バイアス） | 20走分の統計 |

### 2.2 ベイズ推定と動的重み調整

各ファクターのスコアは、**データの信頼度**に応じて補正される。

```
信頼度 = min(実際のデータ数 / 統計的に必要なデータ数, 1.0)

最終スコア = 事前分布（デフォルト値） × (1 - 信頼度) + 観測スコア × 信頼度
```

例：コース適性で1走しかデータがなく、必要数が3走の場合、信頼度=0.33。スコアの67%はデフォルト値（45点）、33%が実際の計算値となる。これにより、データが少ない馬の極端なスコアを防ぐ。

さらに、各ファクターの重みに信頼度を乗じる（最低20%を保証: `重み × max(0.2, 信頼度)`）。削減された分の重みは、信頼度の高いファクターに比例再配分する（**動的重み調整**）。データの多い馬では多面的に、データの少ない馬では信頼できる要素に集中して評価する。

### 2.3 ペースボーナスと加重合計

```
totalScore = Σ (各ファクタースコア × 動的調整後の重み) + ペースボーナス
```

ペースボーナスは脚質と距離の相性に応じて加減算される（短距離=逃げ先行有利、長距離=差し追込有利）。

### 2.4 当日馬場バイアス補正

`track-bias.ts` で同日・同競馬場の結果確定済みレース（3R以上）を分析し、馬場バイアスを算出する。

| バイアス | 算出方法 | 調整幅 |
|---|---|---|
| 枠順バイアス | 上位3着の内枠/外枠比率 → 期待値(0.5)からの乖離 | ±2点 |
| 脚質バイアス | 勝ち馬の第1コーナー通過順位 → 前残り/差し有利判定 | ±2点 |

信頼度（3R=0.375 〜 8R+=1.0）でスケールし、最大 ±4点の調整を適用。

**午後の予想再生成**で効果を発揮する。朝の初回予想生成時は結果確定レースがないためスキップされる。

### 2.5 期待値(EV)に基づくバリューベット検出

Softmax確率推定で各馬の推定勝率を算出し、実際のオッズと掛け合わせて期待値を計算する。

```
P(win_i) = e^(score_i / T) / Σ e^(score_j / T)    (T=8)
EV_i = P(win_i) × odds_i
```

EV > 1.15 の馬を「バリューベット」として推奨。推奨馬券はEV降順でソートされる。

### 2.6 信頼度 (Confidence) の算出

各予想には0〜100%の信頼度が付与される。以下の3要素の合計で算出する。

| 要素 | 最大配点 | 内容 |
|---|---|---|
| スコア差 | 40点 | 上位馬間のスコア差が大きいほど本命が明確→高信頼度 |
| データ充実度 | 35点 | 全馬の平均データ量（過去走数等）が多いほど高信頼度 |
| 統計コンテキスト | 15点 | コース統計・血統統計の充実度 |

## 3. XGBoost MLブレンド

### 3.1 目的

加重合計は線形結合であるため、「コース適性が高い＋距離適性も高い」場合の相乗効果のような非線形パターンを捉えられない。XGBoostの勾配ブースティング木で勝率・複勝率を学習し、加重合計とブレンドすることで精度向上を狙う。

### 3.2 特徴量 (29次元)

| グループ | 特徴量 | 次元数 |
|---|---|---|
| 16ファクタースコア | recentForm, courseAptitude, ... seasonalPattern | 16 |
| コンテキスト特徴量 | 頭数, オッズ, 人気, 年齢, 性別, 斤量, 枠番 | 7 |
| カテゴリエンコード | グレード, 芝/ダート, 馬場状態 | 3 |
| 派生特徴量 | log(1+オッズ), 人気/頭数 | 2 |
| **合計** | | **29** |

### 3.3 推論フロー（TypeScriptネイティブ実装）

```
prediction-engine.ts
  │
  ├─ buildMLFeatures() で29次元ベクトル構築
  │
  ├─ callMLPredict() → TypeScriptでXGBoost JSONモデルを直接走査
  │     ├─ model/xgb_win.json → 全決定木のリーフ値を合算 → sigmoid → 勝率
  │     └─ model/xgb_place.json → 同上 → 複勝率
  │
  └─ ブレンド: 最終スコア = 加重合計 × 60% + ML勝率 × 100 × 40%
     (比率は ML_BLEND_WEIGHT 環境変数で調整可能、デフォルト0.40)
```

Python不要 — XGBoostのJSON決定木形式をTypeScriptで直接パースし、ツリー走査で推論する。これによりVercel Hobbyプランの500MB制限を回避。

**フォールバック**: モデル未配置・パースエラー → MLスキップ、加重合計のみ使用。

### 3.4 学習フロー（GitHub Actions 自動化）

```
GitHub Actions (毎週月曜 12:00 JST / 手動実行可能)
  │
  ├─ Python 3.11 + xgboost をインストール (GitHub側のみ)
  ├─ GET /api/ml-export ← 過去の予想スコア + 実際の着順
  ├─ サンプル100件未満 → スキップ (無駄なコミット防止)
  ├─ 時系列分割 (85% 学習 / 15% 検証)
  ├─ XGBClassifier × 2 (勝利 / 複勝) 学習
  ├─ AUC評価
  └─ model/*.json をコミット・push
       │
       ▼
  Vercel自動デプロイ → 次回予想からML適用
```

手動学習も可能: `notebooks/train_xgboost.ipynb` をGoogle Colabで実行。

## 4. Gemini テキスト強化

### 4.1 目的

数値スコアとテンプレートから生成される予想テキストは、「直近成績が優秀（スコア85）」のような無機質な表現になりがちである。Gemini 2.0 Flashでスコアデータを専門家風の解説に変換し、読み手の納得感を高める。

### 4.2 処理フロー

```
予想エンジンの出力（数値スコア + テンプレートテキスト）
  │
  ▼
Gemini 2.0 Flash API (JSON mode)
  ├─ 入力: レース条件 + 上位6頭のスコア・脚質・血統情報
  ├─ 指示: 競馬プロ解説者として分析テキストを生成
  └─ 出力 (JSON):
       ├─ summary: 総合分析 (2〜4段落)
       ├─ picks[].reasons: 馬ごとの推奨理由 (2〜3個)
       ├─ trackBias: 馬場バイアス分析
       ├─ paceAnalysis: ペース展開予想
       ├─ keyFactors: 重要ファクター (3個)
       ├─ riskFactors: 波乱要素 (2個)
       └─ betReasonings: 馬券種別ごとの根拠
```

**フォールバック**: GEMINI_API_KEY未設定・APIエラー・タイムアウト → 全てテンプレートテキストがそのまま使用される。ランキングや数値は一切変更されない。

## 5. 的中率評価と自動キャリブレーション

### 5.1 評価項目

レース確定後、予想と実結果を自動照合して以下を計測する。

| 指標 | 内容 |
|---|---|
| 単勝的中率 | 本命馬が1着になった割合 |
| 複勝的中率 | 本命馬が3着以内に入った割合 |
| Top3カバー率 | 推奨上位3頭のうち、実際に3着以内に入った頭数 |
| 馬券ROI | 推奨馬券の回収率 |
| 信頼度キャリブレーション | 「信頼度70%の予想が実際に70%当たるか」の検証 |

### 5.2 自動キャリブレーション

評価データが **20レース以上**蓄積されると、各ファクターの重みを自動調整する。

1. 的中レースで高スコアだったファクターの重みを上げる
2. 外れレースで誤誘導したファクターの重みを下げる
3. 算出された新しい重みを自動適用
4. `calibration_weights`テーブルに履歴を保存

## 6. 管理機能

### 6.1 管理画面 (`/admin`)

| 機能 | 説明 |
|---|---|
| フル同期 | 指定日のレース一覧→出馬表→馬詳細→オッズ→予想を一括実行 |
| 予想再生成 | 結果取得→バイアス算出→予想再生成を1ボタンで実行 |
| バルクインポート | 日付範囲指定で過去データを一括取込（チャンク分割） |
| 的中率ダッシュボード | 単勝/複勝的中率、ROI、信頼度帯別精度を表示 |
| 一括照合 | 未照合レースの予想vs結果を一括評価 |
| ウェイト校正 | ファクター重みの自動キャリブレーション |

### 6.2 同期進捗バナー

全ページ共通のヘッダー下に `SyncStatusBanner` コンポーネントを配置。バックグラウンドで同期処理が実行中の場合、5秒ごとにポーリングして進捗を表示する。完了時は緑バナーで8秒間通知。SYNC_KEY は localStorage に永続化し、どのページからでも参照可能。

### 6.3 予想再生成の最適化

「再生成」ボタンは以下の最適化を行い、処理時間を短縮している:

1. **発走時刻を過ぎたレースのみ結果取得**（未発走レースはスキップ）
2. **結果が空の場合はrate limit待ちを省略**
3. **結果確定済みレースは予想再生成をスキップ**（変化しないため）

### 7.3 エラーハンドリング方針

本システムは**段階的フォールバック**を採用している。各レイヤーが独立して失敗してもシステム全体は動作し続ける。

| 障害 | 影響範囲 | フォールバック動作 |
|---|---|---|
| MLモデル未配置 | ML推論 | 加重合計スコアのみで予想（ML導入前と同等） |
| MLモデルパースエラー | ML推論 | 同上 |
| 当日バイアスデータ不足 | バイアス補正 | 補正なし（加重合計のまま） |
| Gemini APIキー未設定 | テキスト生成 | テンプレートベースの分析テキスト |
| Gemini APIエラー | テキスト生成 | 同上 |
| 馬の過去データ不足 | スコア精度 | ベイズ推定で事前分布にフォールバック + 動的重み再配分 |
| スクレイピング失敗 | データ取得 | リトライ (最大2回)、失敗時はスキップして次の処理へ |
| 同期処理タイムアウト | バックグラウンド処理 | 2分後に自動リセット、再実行可能 |
