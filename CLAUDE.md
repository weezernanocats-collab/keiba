# CLAUDE.md

## 改善提案のルール

改善提案を出す前に、必ず現状データの分析を行うこと。

1. 提案の前にボトルネックをデータで確認する（SHAP、的中率内訳、ROI分析など）
2. 「やれること」ではなく「やるべきこと」を提案する
3. 根拠が薄いときは正直に「これは仮説レベル」と明示する
4. 数より質 — 1つの確実な改善 > 5つの思いつき

## モデル改善の前提分析（必須・最優先）

特徴量追加・仮説検証・モデル変更に着手する**前に**、以下を必ず先に実施して提示すること。これを怠ると、市場が既に織り込んでいる情報の二重計上に時間を浪費する。

1. **AI単独実力の計測**: オッズ・人気関連特徴量を全て除外したモデルのTop-1/ROIを測定し、市場ベースライン（1番人気の的中率）と比較する
2. **市場依存度の定量化**: 全特徴量モデル vs オッズ除外モデルの差分 = 市場コピー分。AI独自の付加価値（= 全特徴量 - 市場ベースライン）を算出する
3. **上記を踏まえた改善方針の判断**: AI付加価値がゼロまたはマイナスなら、特徴量追加ではなくモデル構造・損失関数・データソースの見直しが先。「市場が知らない情報は何か」を特定してから仮説を立てる

## データ保全（最重要）

### 原則: 意図しないデータ削除を防ぐ

必要な削除（レース単位の予想上書き等）はOK。問題は**操作対象外のデータまで巻き込む削除**。

- **DELETE文はWHERE句で操作対象を明確に限定する。** 条件なしの `DELETE FROM テーブル` は原則使わない。
- **再生成系スクリプトは対象範囲を指定必須にする。** 例: `--regen` は `--date` と併用必須。
- **スクリプト新規作成・改修時は、既存の全件削除パターンを残さない。** 日付・ID等で範囲を絞る設計にする。
- **大量削除の前にSELECT COUNTで影響範囲を確認する。**
- **大量データの削除が必要な場合は、ユーザーに影響範囲を提示して許可を得てから実行する。**

### 既知の危険スクリプト

| スクリプト | リスク | 対策 |
|---|---|---|
| `scripts/regen-predictions.ts` | 全predictions+results無条件削除 | 使用禁止。gen-predictions-optimized.ts --date --regenを使う |
| `src/lib/bulk-importer.ts:clearAllData()` | 全テーブル完全クリア | clearExisting=trueの呼び出しに注意 |
| `scripts/repair-roi.ts` | 全prediction_results無条件削除 | 日付範囲指定の追加を検討 |

### 安全な削除パターン

- race_id指定で1レースの予想を上書き → OK
- horse_id指定でpast_performancesを再取得 → OK
- 日付指定で特定日の予想を再生成 → OK

## データリーケージ防止

予測に使う全クエリ（getHorsePastPerformances, getJockeyStats, buildRaceContext等）は必ず `beforeDate` フィルタを適用すること。レース日以降のデータが予測に混入すると、見かけ上の精度が上がるが本番では機能しない。

## netkeiba レート制限

約2,600リクエストでnetkeiba側から400エラーになり、12時間以上アクセス不可になる。スクレイピング時は `rateLimitMs` を守り、大量取得時はチャンク分割すること。

## 改修時のチェックリスト

コードを変更する前に以下を確認する:

1. **データ削除を伴うか？** → WHERE句で範囲限定されているか確認
2. **スケール・フォーマット変更か？** → 既存データとの整合性を確認（例: 信頼度閾値の変更は再生成が必要）
3. **モデル更新か？** → 推論結果が変わるため、必要に応じて予想再生成の計画を立てる
4. **外部APIを呼ぶか？** → レート制限に注意（netkeiba: 2,600req上限）

## デプロイ前チェックリスト（必須）

本番デプロイ前に以下を全て実施すること。2026-03-21 障害の再発防止策。

### クエリ変更時
- [ ] **本番データでのクエリ性能テスト**: 変更したSQLを本番DBに対してローカルで実行し、応答時間を確認（目安: 5秒以内）
- [ ] **相関サブクエリの回避**: `WHERE x = (SELECT ... WHERE ...)` パターンは本番データ量で爆発する。JOINまたはアプリ側で処理する
- [ ] **大量行を返すクエリの確認**: JOINで行数が爆発しないか（predictions × race_entries = 80,000+行）

### デプロイ手順
- [ ] **段階的デプロイ**: 複数ファイル変更時は1-2ファイルずつデプロイし、各段階で動作確認
- [ ] **デプロイ後の即時確認**: `/api/races?date=今日` と `/api/predictions/{raceId}` を叩いて200を確認
- [ ] **「影響ゼロ」は禁句**: 影響範囲を具体的に列挙し、各項目の確認結果を提示する

### Turso / インフラ
- [ ] **Turso接続プロトコル**: HTTPS (`https://`) を使用。`libsql://` (WebSocket) は障害リスクあり
- [ ] **batch() の使用禁止**: `db.batch()` はTurso障害でハングする。sequential execute を使う
- [ ] **トークン更新時**: Vercelプロジェクト（keiba）+ GitHub Secrets + .env.local を全て同時更新
